  - Persisted-actor validation for admin actions
  - Reusable authorization dependencies replacing route-local checks
  - Explicit `401` vs `403` behavior and tenant fail-closed semantics
- Security and identity foundations are coherent:

  - Stable `google_sub` binding protections added
  - Active/inactive lifecycle gates enforced consistently at auth boundaries
- Quality posture improved story-to-story:
  - API regression suite expanded significantly (final evidence shows `53 passed`)
  - Repeated review findings were addressed and revalidated in same story cycles
- Auditability and policy controls are in place for Epic 2 dependencies:
  - Audit event capture for sensitive admin mutations
  - Versioned organization safety-controls defaults with controlled read/write access

## Challenges and Growth Areas
- Story 1.1 exposed environment fragility:
  - Dependency installation failed in constrained network conditions (`EAI_AGAIN`) while other validations passed.
  - This created ambiguity around “done” versus “operationally reproducible.”
- Process consistency issues appeared in early story artifacts:
  - Some story status metadata drifted from actual workflow state.
  - File-list tracking occasionally missed changed files until review feedback corrected it.
- Test execution reliability needed hardening:
  - API harness hang required refactor to stable `httpx.ASGITransport` pattern.

## Key Insights and Lessons Learned
- Enforce persisted identity/role/org checks in one reusable dependency layer, not in individual routes.
- Keep story metadata and file lists strict; documentation drift slows review and increases rework.
- Treat runtime/test harness stability as a product quality concern, not just a local tooling issue.
- Review-driven tightening worked well: high/medium findings were systematically converted into durable guardrails.
