## Successes and Strengths

- **Full pipeline delivered end-to-end**: lesson generation → artifact capture → async AI grading → teacher approval → weakness recommendations. The workflow is coherent and complete.
- **Centralized auth dependency — zero drift**: `app.domains.auth.dependencies.require_authenticated_actor` used on every protected endpoint across all 6 stories. No route-local authorization found in any code review. This was an explicit Epic 1 action item — fully delivered.
- **Standard async test harness established and held**: `httpx.ASGITransport` + `asyncio.run(scenario())` pattern adopted in Story 2.3 and used consistently through 2.6. The discovery that BackgroundTasks run synchronously in ASGITransport tests eliminated all async timing complexity in 2.4–2.6.
- **Class-boundary lesson applied immediately**: Story 2.2 [HIGH] finding (missing class-boundary validation for student variants) was learned and applied proactively in 2.3, 2.4, 2.5, and 2.6 — never repeated across the remaining 4 stories.
- **Domain boundary discipline**: The `grading` domain never imported from `courses` or any other domain across 6 stories of additive development.
- **Epic 3 visibility gates ready**: `is_approved` (Story 2.5) and `is_confirmed` (Story 2.6) flags implemented cleanly. `student_id` denormalized on all recommendation records. Epic 3 parent/student progress endpoints can build directly on these without rework.
- **Test suite growth**: 6 tests (post-2.1) → 71 → 116 → 137 → ~166 → 189+ tests at epic close. 100% AC mapping maintained across all stories.
- **Review-driven tightening**: Every story captured HIGH and MEDIUM review findings and resolved them systematically — same pattern that worked in Epic 1.

## Challenges and Growth Areas

- **Missing auth tests recurred across 4/6 stories**: The pattern of missing unauthenticated (401) and non-teacher (403) tests for new endpoints appeared as a review finding in 2.1 (Medium), 2.2 (HIGH), 2.3 (HIGH), and 2.6 (Medium). The fix is always fast but the rework cost compounds across stories. Root cause: auth tests are added reactively after review feedback, not built into the initial implementation checklist.
- **File list tracking gaps in 2/6 stories**: Story file lists drifted from actual git-changed files in 2.2 (MEDIUM) and 2.6 (Medium). Adds noise to every review and reduces traceability.
- **Attempt count double-increment (Story 2.6)**: `process_recommendation_job` used the stale `rec_job.attempt_count` in the second `update_recommendation_job` call instead of `updated_rec_job.attempt_count`. Subtle off-by-one caught in code review. Indicates service methods with multiple sequential mutations need extra care.
- **Known deferred debt**: InMemoryCoursesRepository thread safety (acknowledged 2.2), response envelope deviation from architecture spec (acknowledged 2.2), stub AI/S3/queue — all intentional trade-offs for the in-memory phase but will require dedicated stories before production.
