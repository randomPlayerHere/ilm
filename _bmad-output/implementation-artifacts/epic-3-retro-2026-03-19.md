- **`_require_parent_or_student` guard — write once, reuse everywhere**: Story 3.1 established the access control pattern. Stories 3.2, 3.3, and 3.4 reused it untouched. Zero drift across 4 stories of additive development.
- **Learning-in-action within the epic**: Story 3.2's code review hardened ISO 8601 sort to use `datetime.fromisoformat()`. By Story 3.3, the team proactively applied this pattern with secondary sort keys — no review needed to catch it.
- **Domain boundary discipline held**: The `progress` domain never imported from `grading.service`, `grading.router`, or `students.router` across all 4 stories. The `notifications` domain was cleanly isolated from day one.
- **Clean scope boundary on Story 3.4**: Preference persistence only — no delivery engine, no workers, no notification dispatch. Epic 4 can build the delivery layer without touching 3.4's code.
- **Test suite growth**: 237 → 263 tests (+26) with zero regressions across all 4 stories.
- **Code review effectiveness**: Every story went through 1–2 review cycles. All findings resolved before done status. The review process is the reliable safety net.
- **NFR perf test self-correction**: Added reactively in reviews for 3.2 and 3.3, but by 3.4 the dev agent included it in the initial implementation. Pattern learned within the epic.
