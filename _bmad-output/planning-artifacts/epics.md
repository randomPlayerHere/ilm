# ilm - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for ilm, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

## Requirements Inventory

FR2: Users can sign in using Google sign-in.
FR3: Admins can create and manage organizations.
FR4: Admins can invite, activate, and deactivate users.

FR6: Admins can manage user-to-organization memberships.

### NonFunctional Requirements

NFR1: 95% of authenticated screen loads in the mobile app (teacher, parent, student) and companion web interface (principal, admin) must complete within 2.0 seconds under normal operating load.
NFR2: 95% of core interactive actions (open student profile, open class analytics, open message thread) must complete within 1.5 seconds under normal load.
NFR3: 95% of teacher message send actions must complete within 1.0 second, excluding downstream notification delivery latency.
NFR4: 95% of AI grading job submissions must acknowledge receipt within 3 seconds.
NFR5: 90% of AI grading jobs for single assignment submissions must produce reviewable output within 60 seconds.
NFR6: 100% of protected data in transit must use TLS 1.2+ as verified by automated monthly TLS endpoint scans and zero allowed critical TLS findings.
NFR7: 100% of protected data at rest must be encrypted with managed keys as verified by quarterly storage encryption audits across all production data stores.
NFR8: 100% of protected data-access requests must enforce tenant isolation checks, and cross-tenant read/write attempts must fail closed in automated authorization tests and production monitoring.
NFR9: 100% of protected server-side operations must perform role-based authorization checks, validated by automated integration tests in CI on every release.
NFR10: The system must record immutable audit events for grade changes, role changes, and sensitive data-access events within 30 seconds of the triggering action.
NFR11: The system must enforce data minimization so no non-required student/guardian fields are collected in MVP workflows, as verified by quarterly privacy data-inventory reviews.
NFR12: The platform must provide COPPA/FERPA-aligned parental access, correction, and disclosure-request workflows with request completion within 30 calendar days and auditable request logs.
NFR13: Monthly service availability for core teacher/parent/student workflows must be at least 99.5%, excluding scheduled maintenance windows.
NFR14: No single infrastructure failure may cause loss of committed grade records or approved feedback.
NFR15: At least 99% of failed critical asynchronous jobs (grading pipeline and notification dispatch) must be retried within 5 minutes and processed idempotently with zero duplicate side effects.
NFR16: Backup and restore procedures must support restoration of core production data within 4 hours for high-severity incidents.
NFR17: The architecture must support at least 10x growth from initial pilot volume without full re-architecture of core services.
NFR18: The platform must horizontally scale stateless application services and async workers to sustain 10x pilot baseline throughput with no more than 20% p95 latency degradation during quarterly load tests.
NFR19: Data-access patterns must remain tenant-safe under load, with zero confirmed cross-tenant data exposure incidents in stress tests and production monitoring.
NFR20: All MVP workflows across mobile app and companion web interface must conform to WCAG 2.1 Level AA, validated by automated accessibility scans and manual accessibility audit before each production release.
NFR21: Core user journeys on the companion web interface (admin user management, principal dashboards) must be fully keyboard accessible. Mobile app journeys must follow platform accessibility guidelines (iOS/Android).
NFR22: 100% of production UI screens across mobile app and companion web interface in MVP scope must pass WCAG 2.1 AA checks for color contrast, focus indicators, and semantic labeling in release-gate accessibility audits.
NFR23: External integration failures (e.g., Google Classroom when enabled) must not block core in-app grading, progress viewing, or messaging workflows.
NFR24: Integration sync operations must emit structured logs (request_id, tenant_id, connector_id, status, error_code) and trigger failure alerts within 5 minutes of repeated sync failure conditions.
NFR25: All integration contracts must be versioned, and connector updates must pass backward-compatibility contract tests for at least one prior supported version before release.
NFR26: 100% of production services must emit structured logs and core service metrics, and at least 95% of inbound requests must include distributed traces, sufficient to diagnose Sev-1/Sev-2 incidents within incident response SLAs.
NFR27: Alerting must detect sustained increases in error rate, latency, and failed async jobs within 5 minutes.
NFR28: Security-relevant events (failed auth spikes, permission-denied anomalies, unusual sensitive access patterns) must be monitored and alertable.

### Additional Requirements

**Architecture — Starter Template & Monorepo:**
- Initialize repository with Turborepo using `npx create-turbo@latest`; this should be the first implementation story.
- Monorepo structure: `apps/mobile` (React Native/Expo), `apps/admin-web` (React SPA/Vite), `apps/api` (FastAPI/Python).
- Shared packages: `packages/contracts` (OpenAPI types), `packages/design-tokens`, `packages/ui` (shared Tamagui components).

**Architecture — Backend:**
- FastAPI (Python 3.12+) as core API layer with domain modules (`/auth`, `/courses`, `/grading`, `/students`, `/messages`, `/notifications`, `/org-analytics`, `/admin`, `/onboarding`).
- Worker service (FastAPI/Celery or ARQ) for async AI grading, notification dispatch, and digest assembly jobs.
- PostgreSQL 16.x (RDS) + RDS Proxy for connection pooling.
- Alembic (SQLAlchemy) for database migrations.
- REST + OpenAPI 3.1 contracts versioned under `/v1`.
- JWT auth with refresh token rotation, per-device session revocation.
- Centralized tenant/org isolation policy module on all protected queries.
- Immutable grade_versions and audit_events tables.

**Architecture — Infrastructure & Deployment (AWS):**
- ECS Fargate: separate services for API and worker (independent scaling).
- S3 for assignment images (pre-signed URL upload), ECR for container images.
- SQS for async job queues (grading, notifications) with DLQ.
- CloudFront unified edge for API + web (WAF, TLS, DDoS baseline via Shield Standard).
- Secrets Manager for all secrets (no secrets in env vars or Docker images).
- Route 53 (DNS), ACM (TLS certificates), CloudWatch (logs/metrics/alarms).
- Environment tiers: dev, staging, prod with isolated databases.
- CI/CD: GitHub Actions (lint → test → build → push to ECR → deploy to ECS; staging → manual gate → prod).

**Architecture — Mobile:**
- React Native (Expo) with Tamagui design system.
- Expo EAS Build for automated iOS (TestFlight) and Android (Play Store internal) distribution.
- App store submission pipeline from sprint 1 (Apple provisioning profiles, signing certs, Play Console).
- Push notifications via FCM (Android) + APNs (iOS), unified via SQS-backed backend dispatch.
- Offline photo queue (AsyncStorage/SQLite) with auto-upload on reconnect.
- Cached dashboard views for offline reading.
- Client-side image compression (~2MP) via expo-image-manipulator.
- Role-scoped bottom tab navigation (Teacher: 5 tabs + camera FAB; Parent: 4 tabs; Student: 4 tabs).

**Architecture — Companion Web:**
- React SPA (Vite + Tamagui) — static deploy on S3 + CloudFront, no SSR.
- Sidebar navigation for principal/admin roles.
- Data-dense tables, charts, and filter-heavy interfaces.

**Architecture — Security & Compliance:**
- JWT auth, RBAC, tenant isolation on every protected query.
- Signed URL upload/download for student work images.
- Audit logs for grade edits, role changes, sensitive data access, consent status changes.
- COPPA consent confirmation gate: admin verifies consent before activating under-13 accounts.
- App store compliance: education-institution positioning (not Kids category); SDK audit per release.
- Privacy nutrition labels (Apple) and Data Safety sections (Google) maintained.

**Architecture — Observability:**
- Structured JSON logs with correlation/request IDs across API and workers.
- CloudWatch alarms for p95 latency breaches, 5xx rate, job failure rate, SQS DLQ depth.
- 95%+ distributed traces on requests.
- Security event monitoring: failed auth spikes, permission-denied anomalies.

**UX — Design System:**
- Tamagui with shared design tokens (colors, typography, spacing, radii, shadows).
- "Clear Path" design direction: confident minimalism with purposeful color.
- Warm growth palette: greens/golds for progress, amber for focus areas, terracotta for errors. No bright red for below-average.
- Inter font family throughout; JetBrains Mono for numerical data.
- 44px minimum touch targets on mobile.
- Skeleton loading states for all data-dependent screens.

**UX — Core Components:**
- `TrendIndicator`: hero up/down/steady arrow with one-sentence explanation (parent/student dashboards).
- `GradingCard`: photo + AI suggestion + score adjuster + feedback editor + confidence badge + rubric + approve action.
- `GrowthTimeline`: longitudinal progress visualization with growth framing (never "weaknesses" — always "Focus" or "Areas for growth").
- `ContextualMessageComposer`: message input pre-linked to student/assignment context.
- `OnboardingWizard`: role-specific step-through with progress indicator.
- `SkeletonLoader`: content-shaped loading placeholders.

**UX — Interaction Patterns:**
- Camera-to-grade pipeline: one tap camera → snap → AI suggest → review → approve (under 2 minutes per assignment).
- Parent invite flow: tap invite link → Google sign-in → child pre-linked → dashboard visible.
- Progressive disclosure: answer first, details on demand.
- Contextual messaging: messages auto-linked to student + assignment context.
- AI positioned as "assistant" language: "Suggested," "Draft," "Your call" — never "Determined" or "System score."
- Growth framing for students: "You improved in X" not "You failed at Y."
- Default notification cadence: weekly digest, not instant everything.

**UX — Accessibility:**
- WCAG 2.1 AA compliance across mobile and web.
- Color tokens validated for contrast ratios at token level.
- Screen reader labels on all custom components.
- Reduced motion support (respects OS setting).
- Focus indicators: 2px primary outline on all focusable elements.
- No information conveyed only through color — always paired with icons/labels/patterns.

**UX — Offline & Connectivity:**
- Offline photo capture: saved to local queue with "Pending" badge, auto-uploads on reconnect.
- Dashboard viewing: cached data shown with "Last updated: [timestamp]" banner.
- Messaging: queued with "Sending..." status, auto-sends on reconnect.
- Sign-in requires network — not available offline.

I want the repository restructured as a Turborepo monorepo with mobile, admin-web, and API workspaces plus shared packages,
So that all teams can develop, build, and share code across surfaces from a single repo.

**When** the monorepo scaffold is initialized using Turborepo
**Then** the repo contains `apps/mobile` (Expo/React Native), `apps/admin-web` (Vite/React), and `apps/api` (existing FastAPI) workspaces
**And** shared packages exist: `packages/contracts`, `packages/design-tokens`, `packages/ui`

**Given** the monorepo is scaffolded
**When** a developer runs the dev command
**Then** all three app workspaces start concurrently via Turborepo task orchestration
**And** changes to shared packages propagate to consuming apps

**Given** the mobile workspace is initialized
**When** the Expo project is configured
**Then** Tamagui design system is installed with base theme tokens (colors, typography, spacing from UX spec)
**And** TypeScript is configured across all workspaces with shared tsconfig base

As an admin,
I want to confirm that parental consent has been collected before activating accounts for children under 13,
So that the platform meets COPPA compliance requirements.

**Acceptance Criteria:**

**Given** a student is flagged as under 13 (based on grade level or explicit age indicator)
**When** an admin has not confirmed parental consent for that student
**Then** the parent and student accounts associated with that child remain inactive
**And** those accounts cannot access any protected data or features

**Given** an admin views a student requiring consent confirmation
**When** they confirm that parental consent has been collected (per the school's process)
**Then** the consent status is recorded with admin identity and timestamp
**And** the associated parent and student accounts are activated

**Given** consent has been confirmed
**When** the consent record is queried
**Then** the confirmation is auditable with actor, timestamp, and student identity
**And** the consent status is visible in the admin's student management view
